Part I — Project Overview, Objectives & Features

1.1 What This Project Is

The Student Grade Management System is a console (terminal) Java application that lets a school manage students, subjects, and grades, and derive reports and analytics from that data: GPA on a 4.0 scale, class-wide statistics, bulk CSV grade import, exported text reports, and student search. It began as a smaller Lab 1 exercise (a flat, 9-class in-memory design) and was extended for this assessment ("v2") into a layered, testable, SOLID-compliant application with a real build (Maven), a real test suite (JUnit 5 + Mockito, 377 tests), a code-coverage and static-analysis toolchain (JaCoCo + SonarQube), and a real CI/CD pipeline (GitHub Actions).

There is no external database and no network layer. All data — students, subjects, grades — lives in fixed-size in-memory arrays for the lifetime of the process and is lost when the program exits. This was a deliberate constraint of the assignment brief, not an oversight, and is discussed candidly in Part XII.

2.2 Why It's Organized This Way — Layer by Layer

This is a classic layered architecture: each layer only talks to the layer directly beneath it, and only through that layer's interface, never its concrete class. Requests flow strictly downward; nothing in a lower layer ever calls back up into a higher one.

2.3 A Design Decision Worth Explaining: Colocated Interfaces

Every interface in this codebase sits in the same package as its one implementer — service/StudentService.java next to service/StudentServiceImpl.java, calculators/Calculable.java next to calculators/GPACalculator.java, and so on. This wasn't always true: early on there was a separate interfaces/ package holding Calculable, Exportable, and Searchable while service/repository already colocated theirs — two different conventions for the identical relationship. It was unified during a professional-structure review (see Part X's commit history) because a class implementing an interface isn't "a kind of interface" — it belongs with its own feature area, not filed away by which contract it happens to satisfy.

2.4 Why Fixed-Size Arrays, Not a HashMap or a Database

The assignment brief explicitly required in-memory, array-based storage (a carry-over constraint from the original Lab 1 spec). Each repository owns exactly one primitive array (Student[50], Grade[200], Subject[50]) plus a running count of how many slots are filled. A linear scan (for loop) is used for lookups — O(n), not O(1) like a HashMap would give — which is an accepted, deliberate trade-off given the array-size ceiling (at most 200 grades) makes the performance difference immaterial in practice.

Part III — Object-Oriented Programming Concepts Applied

3.1 Encapsulation

Every field in every domain class is private; access is only through explicit getters/setters or behavior methods. Student.grades (a List<Double>) is never exposed for direct external mutation except through addGrade(double)/setGrade(double) — callers cannot reach in and corrupt the list's internal state without going through a method that owns the invariant.

public abstract class Student {
    private String studentId;         // no caller can see or set this except through a method
    private List<Double> grades = new ArrayList<>();

    public List<Double> getGrades() { return grades; }   // read access
    public void addGrade(double grade) { grades.add(grade); } // controlled write
}
Honest caveat: getGrades() returns the live, mutable List reference rather than an unmodifiable copy — a determined caller could call .clear() on the returned list from outside the class. In practice only StudentManager.hydrateGrades() does this (deliberately, to refresh the transient list from the real source of truth in GradeRepository), but a stricter version of encapsulation would return Collections.unmodifiableList(grades) or a defensive copy. Good to know as a discussion point if asked "is this fully encapsulated?"

3.2 Inheritance

Two parallel class hierarchies model "is-a" relationships:

HierarchyBase (abstract)SubclassesWhat differs per subclass
StudentsStudentRegularStudent, HonorsStudentPassing grade threshold (50% vs 60%), extra checkHonorsEligibility() behavior, own StudentType value
SubjectsSubjectCoreSubject, ElectiveSubjectDisplay text, SubjectType value, an isMandatory() flag

Shared state and behavior (student ID, name, age, grade list, calculateAverageGrade(), isPassing()) live once in the abstract base class; only what genuinely differs is overridden. This is the textbook argument for inheritance: avoid duplicating the 90% that's identical across subtypes.

3.3 Polymorphism

Every place that holds a Student reference — StudentRepositoryImpl's array, List<Student> returned by getAllStudents(), the parameter of displayStudentDetails() — never needs to know or check whether the actual object is a RegularStudent or HonorsStudent. Calling student.getPassingGrade() or student.displayStudentDetails() runs whichever subclass's override actually applies, decided at runtime by the object's real type — not by the reference's declared type. Under the hood (see Part V) this is dynamic dispatch: the JVM looks up the method in the object's actual class's method table at the moment the call executes, not at compile time.

A second, narrower form of polymorphism used here is pattern-matching instanceof (Java 16+), used sparingly and only where genuinely subclass-specific behavior is needed with no base-class equivalent:

if (student instanceof HonorsStudent hs) {
    hs.checkHonorsEligibility(); // only HonorsStudent has this method
}

This is deliberately not used for classifying "which type is this student" (that would fight polymorphism) — see Part IV's OCP discussion for where that exact anti-pattern was found and fixed by introducing Student.getType(): StudentType instead.

3.4 Abstraction

Two abstraction mechanisms are used, deliberately for different purposes:

  • Abstract classes (Student, Subject) — used when subclasses share real, inherited state (fields) as well as behavior. A subclass can only extend one abstract class.
  • Interfaces (Calculable, Exportable, Searchable, Gradable, MenuAction, every *Service/*Repository) — used to describe a pure contract with no shared state, so a class can implement many of them at once and different, unrelated classes can satisfy the same contract (e.g. any future report format could implement Exportable without touching ReportGenerator).

Part IV — SOLID Principles: Where and How

The assignment brief names all five explicitly and even suggests example classes. What follows is where each principle is actually demonstrated in this codebase today, including two real violations that were found during a dedicated audit and fixed (not just theoretical box-ticking).

4.1 Single Responsibility Principle (SRP)

Definition: a class should have exactly one reason to change.

ClassIts one job
GPACalculatorPercentage↔GPA↔letter conversion, cumulative GPA, class rank — nothing else
StatisticsCalculatorDescriptive statistics over a grade list — nothing else
ReportGeneratorBuilds report text — knows nothing about files
FileExporterWrites a string to a file — knows nothing about report content
CSVParserParses CSV text into valid rows + error messages — never persists anything
BulkImportServiceOrchestrates parse → persist → log for a bulk import — the actual parsing and persisting are delegated
SRP was violated, found, and fixed — twice: Main.java had grown to 760 lines doing composition-root wiring, console I/O, business orchestration for all 9 menu features, role authorization, and exception-to-message translation — five reasons to change in one file. Fixed by extracting one console.*Action class per feature, each implementing MenuAction. A second, smaller SRP violation then surfaced in what was left: Main still mixed "build the dependency graph" with "run the interactive menu loop," and the loop was bound to System.in in a static final Scanner field, which also made it untestable. Split again into app.Main (composition root only, ~65 lines) and app.ConsoleApp (the loop, taking its Scanner via constructor instead of a static field). This is a genuinely good interview talking point: it shows you can find an SRP violation in real code, not just recite the definition — and that fixing one can reveal a second, smaller one underneath.

4.2 Open/Closed Principle (OCP)

Definition: open for extension, closed for modification — adding new behavior shouldn't require editing existing, working code.

  • Adding a third Subject subtype requires only a new class implementing displaySubjectDetails()/getSubjectType() — no existing subject code changes.
  • LetterGrade and GpaLetterGrade are enums whose thresholds live as data on each constant; adding a new grading tier means adding one enum constant, not editing an if-chain.
OCP was violated, found, and fixed: StudentSearcher.searchByType(boolean isHonors) and StatisticsCalculator.compareStudentTypes() classified students via a boolean parameter and instanceof HonorsStudent checks — even though every Student already carried a StudentType enum internally, just never exposed publicly. Adding a third student type would have meant reworking a boolean parameter (which can only ever represent two states) and every instanceof check by hand. Fix: exposed abstract StudentType Student.getType(); both call sites now compare getType() against the enum instead. Adding a third type is now purely additive: one enum constant, one subclass.

4.3 Liskov Substitution Principle (LSP)

Definition: anywhere a base type is expected, any subtype must work correctly as a drop-in replacement, without surprising callers (no strengthened preconditions, no weakened postconditions, no new exceptions the base didn't promise).

Verified clean: RegularStudent/HonorsStudent and CoreSubject/ElectiveSubject both honor their base contracts exactly — no override throws an exception the base method doesn't declare, no override requires a caller to do anything extra first. Any code holding a Student or Subject reference works identically no matter which concrete subtype it actually holds.

4.4 Interface Segregation Principle (ISP)

Definition: no client should be forced to depend on methods it doesn't use — prefer several small, focused interfaces over one fat one.

Every interface here is small and single-purpose: Calculable (4 methods, all GPA-related), Exportable (2 methods), Searchable (4 methods, all search variants), Gradable (2 methods). No interface forces an implementer to stub out irrelevant methods.

4.5 Dependency Inversion Principle (DIP)

Definition: high-level modules shouldn't depend on low-level modules directly — both should depend on abstractions.

Every layer takes its collaborators as constructor-injected interfaces, never concrete classes:

public class StudentServiceImpl implements StudentService {
    private final StudentRepository studentRepository;  // interface, not StudentRepositoryImpl
    public StudentServiceImpl(StudentRepository studentRepository) {
        this.studentRepository = studentRepository;
    }
}

Main is the single, intentional exception — the composition root. Something in the program has to new the concrete implementations at least once; that's Main's entire job, and only Main does it. Every other class is unaware which concrete implementation it was handed.

Part V — Java Language Mechanics (Behind the JVM Curtain)

This section is written for an engineer, not just "what does this keyword do" — it explains what the JVM actually does with each construct, since that's what distinguishes someone who memorized definitions from someone who understands the runtime.

5.1 JVM Runtime Data Areas — Where Things Actually Live

AreaWhat lives thereIn this codebase
Method Area / MetaspaceClass metadata, method bytecode, static fieldsStudent.studentCounter, Grade.gradeCounter — one copy total, shared by every instance
HeapEvery object created with newEvery Student, Grade, Subject, DTO, exception instance
Stack (one per thread)Local variables, method call frames, return addressesEvery local variable inside e.g. GPACalculator.cumulativeGPA()
PC RegisterAddress of the currently executing instruction, per threadIrrelevant here — this app is single-threaded (one main thread only)

This app never creates a second thread, so there is exactly one call stack and no concurrency concerns to reason about — which matters for the volatile discussion below.

5.2 static — Class-Level State

A static field belongs to the class, not to any instance — there is exactly one copy, allocated once when the class is first loaded by the classloader, living in the Method Area for the lifetime of the JVM process. This codebase uses it for two auto-incrementing ID counters:

public abstract class Student {
    private static int studentCounter = 0;  // ONE counter, shared by every Student ever created

    private String generateStudentId() {
        studentCounter++;
        return String.format("STU%03d", studentCounter);
    }
}

Every new RegularStudent(...) and every new HonorsStudent(...) increments the same counter, because both subclasses inherit the one static field from Student — there's no per-subclass copy. This is exactly why the test suite's own README warns: never assert an exact ID like "STU001", because the counter is shared across every test class in the same JVM run (Surefire may run test classes in any order) — assert the format instead.

Why not an instance field instead? If studentCounter were an instance field, every new Student would start its own counter at 0 — generateStudentId() would always produce "STU001", guaranteeing ID collisions. The whole point of static here is shared state across every instance.

5.3 final — Three Different Meanings

Applied toMeaningExample in this codebase
a variable/fieldCan be assigned exactly once (reference cannot be reseated after that — though a mutable object it points to can still change internally)private final StudentRepository studentRepository; — every repository/service field is final, assigned once in the constructor, guaranteeing it's never accidentally reassigned later
a methodCannot be overridden by a subclassStudent.getStudentType() is final — it always delegates to getType().name(), so no subclass can quietly reintroduce the duplicated-string-literal bug this replaced
a classCannot be subclassed at allNone declared explicitly here, but every DTO/utility class (StudentDTO, DateFormats) is implicitly "final in spirit" — they hold no behavior worth extending

5.4 volatile — Not Used Here, and Why That's Correct

This keyword doesn't appear anywhere in this codebase, and an engineer should be able to explain why that's the right call, not just what the keyword means:

volatile solves a multi-threaded visibility problem: without it, one thread's write to a shared field might sit in that thread's CPU cache and never become visible to another thread reading the same field, because the compiler/CPU are free to reorder and cache reads/writes for performance under the Java Memory Model (JMM). Marking a field volatile forces every read to go to main memory and every write to flush to main memory immediately, establishing a happens-before relationship between threads.

This application has exactly one thread — the JVM's main thread running the console loop. There is no second thread ever created (no Thread, no ExecutorService, no async callback), so there is no visibility problem to solve. Marking studentCounter or useRoleBased volatile would add a small runtime cost (bypassing CPU caching) with zero benefit. If this app ever became a multi-user server handling concurrent requests, that calculus would change immediately — at that point, unsynchronized shared mutable state like Student.studentCounter would become a real race condition, and the fix would likely be AtomicInteger rather than a bare volatile int, since incrementing (counter++) is a read-modify-write that volatile alone does not make atomic.

5.5 this and super

  • this — a reference to the current object instance; used here mainly in constructors to disambiguate a parameter name from a field of the same name (this.studentId = studentId;), and via this(...) to chain constructor overloads, e.g. GradeServiceImpl's no-arg constructor delegates to its 3-argument constructor with default Impls.
  • super — refers to the parent class; used in every subclass constructor to initialize inherited fields (super(name, age, email, phone); in RegularStudent's constructor calls Student's constructor first, before any subclass-specific code runs). Constructor chaining via super(...) is mandatory — Java inserts an implicit super() call if you don't write one, and it must be the very first statement in a constructor.

5.6 enum — What the Compiler Actually Generates

Writing public enum Role { TEACHER, STUDENT } is syntactic sugar. The compiler generates a real class that implicitly extends java.lang.Enum<Role>, and each named constant becomes a public static final Role field — a genuine singleton instance, constructed exactly once when the class is first loaded, never again. That's why == is always safe (and preferred over .equals()) for comparing enum values — there is only ever one Role.TEACHER object in the entire JVM.

public enum GpaLetterGrade {
    A(93, 4.0, "A"), A_MINUS(90, 3.7, "A-"), /* ... */ F(0, 0.0, "F");

    private final double minPercentage, gpaPoints;
    private final String label;

    GpaLetterGrade(double minPercentage, double gpaPoints, String label) { // implicitly private
        this.minPercentage = minPercentage; this.gpaPoints = gpaPoints; this.label = label;
    }
}

Each enum constant can carry its own fields, exactly like an object — GpaLetterGrade uses this to keep percentage threshold, GPA points, and display label together as one source of truth, replacing three separately hand-maintained parallel arrays that had drifted apart from each other in an earlier version (the root cause of the project's headline bug, documented as KI-1 in CHANGELOG.md).

5.7 Interfaces, Dynamic Dispatch, and invokevirtual

When code calls calculable.percentageToGPA(85) where calculable is declared as the Calculable interface type, the compiler doesn't know which class's method will actually run — only GPACalculator implements Calculable here, but in principle any class could. The JVM resolves this at runtime using the invokeinterface (or invokevirtual for abstract-class polymorphism) bytecode instruction: it looks up the object's actual runtime class, finds that class's method table (a per-class array of method pointers built at class-load time), and jumps to the correct method body. This lookup has a small, constant-time cost — noticeably more than a direct invokestatic call to a static method, but the JIT compiler can often inline it away once it observes the call site is monomorphic (always resolves to the same class) at runtime.

5.8 Exceptions — Stack Unwinding and try-with-resources

All 10 custom exceptions in this project extend RuntimeException (via the shared ApplicationException root) — they are unchecked, meaning the compiler does not force a caller to declare or catch them. When one is thrown (e.g. new StudentNotFoundException(...) inside StudentRepositoryImpl.findStudentById()), the JVM does not immediately search for a handler in that method — it unwinds the call stack frame by frame (repository → service → manager → console action → ConsoleApp's loop), popping each method's local stack frame, running any finally blocks encountered along the way, until it finds a stack frame whose try block is covered by a matching catch. In this app that's always ConsoleApp's central switch of catch clauses, ending in catch (ApplicationException e) — deliberately not catch (Exception e), so a truly unexpected bug (a NullPointerException, say) is not silently swallowed and still crashes loudly, which is exactly what you want during development.

try-with-resources (used in FileExporter/BulkImportService for FileWriter, and in CSVParser for Scanner) compiles to an implicit finally block that calls .close() on the resource — guaranteed to run whether the try block completes normally, returns early, or throws. This requires the resource type to implement java.lang.AutoCloseable.

5.9 Generics and Type Erasure

List<Student>, List<Grade>, List<String> are used throughout. At compile time, the generic type parameter enforces that only a Student can go into a List<Student> — but at runtime, due to type erasure, that information is gone: the compiled bytecode just sees List, and the compiler inserts an invisible cast at every read site. This is why you cannot do new T[10] or check if (list instanceof List<Student>) in Java — the generic type parameter simply isn't present at runtime to check against.

5.10 Streams

Used in a few display-mapping spots, e.g. converting a List<Student> search result to a List<StudentDTO>:

List<StudentDTO> resultDtos = results.stream().map(StudentMapper::toDto).collect(Collectors.toList());

This builds a lazy pipeline (stream() does no work by itself), applies toDto to each element only when the terminal operation (collect) is invoked, producing a new list without mutating the original. StudentMapper::toDto is a method reference — shorthand for the lambda s -> StudentMapper.toDto(s).

5.11 Garbage Collection — Why This App Never Frees Memory Manually

When StudentRepositoryImpl.deleteStudent() removes an entry by overwriting its array slot with null (after moving the last element into its place), the Student object that used to be there becomes unreachable — nothing in the program holds a reference to it anymore. The JVM's garbage collector will eventually reclaim that memory automatically; nothing in this codebase (or in Java generally, outside of rare off-heap scenarios) ever calls free() or manages memory by hand. This is a core reason Java was chosen for this kind of application over C/C++: no manual memory-management bugs (use-after-free, double-free) are possible.

Part VI — Logging Architecture

6.1 The Logger Class

public final class Logger {
    public enum Level { DEBUG, INFO, WARN, ERROR }
    private static volatile Level threshold = resolveThreshold();

    public static void debug(String message) { log(Level.DEBUG, message); }
    public static void info(String message)  { log(Level.INFO, message); }
    public static void warn(String message)  { log(Level.WARN, message); }
    public static void error(String message) { log(Level.ERROR, message); }

    private static void log(Level level, String message) {
        if (level.ordinal() < threshold.ordinal()) return;  // below threshold — skip
        System.err.printf("[%s] %-5s %s%n", timestamp(), level, message);
    }
}
Here volatile IS used — on Logger.threshold — because setLevel() lets tests change the logging threshold at runtime, and while this app is single-threaded in practice, the field is written from test code and read from production code potentially across different execution contexts within the same JVM (parallel test execution is a real possibility with Surefire); marking it volatile is the cheap, correct insurance against that, unlike the counters in §5.4 which truly have no such exposure.

Level filtering: levels are ordered by their enum declaration order (DEBUG < INFO < WARN < ERROR), so level.ordinal() < threshold.ordinal() is a cheap integer comparison that skips building/printing a message below the configured threshold — default INFO, configurable via the JVM system property -Dlog.level=DEBUG, or programmatically via Logger.setLevel(...) in tests.

Why System.err, not System.out: the console menu's user-facing output (menus, results, confirmations) goes to System.out; every log line goes to System.err. This means a user (or a script) can redirect one stream without capturing the other — e.g. java -cp target/classes Main 2>debug.log keeps the menu clean on the terminal while every log line still lands in a file.

6.2 Why a Hand-Rolled Logger, Not Log4j/SLF4J

A real framework (SLF4J + Logback, or Log4j2) offers structured logging, log rotation, multiple appenders (file, console, network), and MDC (per-request context). None of that is needed for a single-threaded console app with no persistence and no concurrent requests — pulling in a full framework would be a dependency for a problem this app doesn't have. The hand-rolled Logger gives exactly the four levels and one output stream this app actually uses, with zero external dependencies. If this evolved into a server application, migrating to SLF4J's facade would be the very next thing to do — its Logger interface has almost the same four-method shape already, which would make that migration mechanical.

Part VII — Exception Handling Architecture

7.1 The Hierarchy

RuntimeException  (java.lang)
  └── ApplicationException  (abstract — the ONE common root for everything this app throws on purpose)
        ├── StudentNotFoundException     (studentId, availableIds — lets Main suggest valid IDs)
        ├── StudentValidationException
        ├── SubjectNotFoundException
        ├── SubjectValidationException
        ├── GradeException                (gradeId)
        ├── InvalidGradeException         (attemptedGrade — e.g. entered 150)
        ├── ExportException               (filePath)
        ├── ImportException                (filePath, successfulRows, failedRows)
        ├── InvalidFileFormatException     (filename, lineNumber)
        └── CSVImportException            (whole-file read failure, distinct from a single bad row)

7.2 Why Everything Extends RuntimeException (Unchecked), Deliberately

Java offers checked exceptions (must be declared with throws or caught, enforced by the compiler) and unchecked ones (no such enforcement). This project chose unchecked deliberately: every custom exception here represents a business-rule violation (a bad student ID, an out-of-range grade) that the caller almost never wants to recover from locally — it wants to bubble all the way up to one central handler in ConsoleApp. Checked exceptions would force every intermediate layer (StudentServiceImpl, StudentManager, every console.*Action) to either declare throws StudentNotFoundException on every method signature or catch-and-rethrow pointlessly — pure boilerplate with no benefit, since there's nowhere useful to "handle" it before ConsoleApp.

7.3 No Generic catch (Exception e) — Why That Matters

ConsoleApp's central handler ends in catch (ApplicationException e), never catch (Exception e). This is a direct, explicit requirement in the assignment brief ("No generic Exception catching"), and the reasoning is sound independent of the brief: a generic catch would also swallow a genuine bug — a NullPointerException from a real coding mistake — printing a vague error message and continuing as if nothing happened, hiding the bug instead of surfacing it. Catching only the app's own exception family lets anything truly unexpected crash loudly and visibly, which is far easier to debug.

7.4 Recovery Data Riding Along With the Exception

Several exceptions carry structured data beyond a message string specifically so the top-level handler can be more helpful than "something went wrong":

throw new StudentNotFoundException(
    "Student with ID '" + studentId + "' not found.",
    studentId,

Part VIII — Class-by-Class Reference

Every class in the src tree (excluding tests/), grouped by package, with its purpose and every non-trivial method. Trivial one-line getters/setters are omitted from the method lists below for brevity — assume every private field has a corresponding public getter unless noted.

8.1 model — Domain Objects

Student model.student.Student (abstract)

Base type for every student. Holds ID, name, age, email, phone, status, and a transient grade list.

calculateAverageGrade(): double — sums the transient grades list and divides by its size; returns 0.0 if empty (avoids division by zero).
isPassing(): boolean — calculateAverageGrade() >= getPassingGrade(); delegates the threshold to the subclass.
abstract getType(): StudentType — the OCP fix from Part IV; each subclass returns its own enum constant.
final getStudentType(): String — getType().name(); cannot be overridden, so the two subclasses can no longer duplicate this logic independently.
abstract getPassingGrade(): double / abstract displayStudentDetails(): void — subclass-specific.
static initializeCounter(int highestExistingSequence): void — called once at startup by StudentManager to advance the static ID counter past whatever's already stored, so freshly-added students never collide with pre-seeded ones.

RegularStudent / HonorsStudent model.student

RegularStudent: passing grade 50% (StudentType.REGULAR). HonorsStudent: passing grade 60%, plus checkHonorsEligibility(): boolean, which recomputes (never caches) whether the student's current average clears their own passing threshold — recomputed on every call, not cached, so it's always current after a grade changes.

Subject (abstract) / CoreSubject / ElectiveSubject model.subject

Mirrors the Student hierarchy: Subject holds name + code; each subclass supplies its own displaySubjectDetails(), getSubjectType(), and an isMandatory() flag (true for Core, false for Elective).

Grade model.grade.Grade — implements Gradable

Immutable-in-spirit value once constructed: ID and date are auto-generated in the constructor and never change afterward.

Grade(String studentId, Subject subject, double gradeValue) — validates the value via recordGrade()/validateGrade() (the Gradable contract) before assigning an ID or date; throws InvalidGradeException if out of range 0–100, so an invalid grade never gets an ID at all.
static reconstruct(gradeId, studentId, subject, gradeValue, date): Grade — a private constructor exposed via a static factory, used only when rebuilding a grade already known-valid (e.g. from bulk import), skipping ID/date auto-generation.
getLetterGrade(): LetterGrade — delegates to LetterGrade.fromNumeric(grade).

Enums model.enums

  • Role { TEACHER, STUDENT } — replaces a former raw boolean isTeacher flag.
  • StudentType { REGULAR(50.0), HONORS(60.0) } — carries its own passing-grade threshold.
  • StudentStatus { ACTIVE, INACTIVE, SUSPENDED, GRADUATED }.
  • SubjectType { CORE, ELECTIVE }.
  • LetterGrade { A(85), B(70), C(55), D(40), F(0) } with static fromNumeric(double): LetterGrade — iterates constants in declaration order, returns the first whose threshold the grade clears.
  • GpaLetterGrade — 11 constants (A through F), each carrying percentage threshold, GPA points, and display label together; fromPercentage() and fromGpaPoints() static factories.

8.2 repository — Persistence Layer

StudentRepository / GradeRepository / SubjectRepository (interfaces) + their *Impl

Each interface defines CRUD-shaped operations (addX, findXById/findXByCode, getAllX, deleteX, plus updateStudent). Each Impl is backed by exactly one fixed-size array (Student[50], Grade[200], Subject[50]) plus a running count field. StudentRepositoryImpl and SubjectRepositoryImpl seed themselves with demo data in their constructors (5 students: 3 Regular + 2 Honors; 6 subjects: 3 Core + 3 Elective). deleteX methods swap the last live element into the deleted slot and null out the vacated last slot — an O(1) removal that doesn't leave a gap, at the cost of not preserving insertion order after a delete.

8.3 service — Business Rules

StudentServiceImpl

addStudent(Student): void — runs StudentValidator.validateStudent() first (name/age/email/phone/ID format), logging and rethrowing on rejection; only then persists.
getStudentById(String): Student — throws (a plain RuntimeException, notably not one of the custom hierarchy, if the repository somehow returns null — defensive code for a state the repository's own contract says can't happen, since findStudentById itself throws StudentNotFoundException on a miss).

GradeServiceImpl

Three constructor overloads (telescoping pattern): no-arg (news its own default Impls — a minor, isolated DIP compromise used only in one test), two-repository, and full three-repository. recordGrade() checks the student exists, then the subject exists, before persisting — the grade's own 0–100 range check already happened inside Grade's constructor, so this method doesn't repeat it.

8.4 manager — Facades

StudentManager

findStudent(String): Student (nullable) — catches StudentNotFoundException internally and returns null instead, giving callers a simple null-check API rather than forcing every caller to catch an exception just to check existence.
hydrateGrades(Student): void (private) — the most important private method in the app: clears the student's transient grade list and refills it from GradeManager, since grades live in their own repository, not on the Student object. Called before every read path that needs an accurate average.

GradeManager

Wraps GradeService and SubjectRepository; adds grade-history formatting (viewGradesByStudent, sorted by date descending, ties broken by grade ID descending) and average calculations split by subject type (calculateCoreAverage, calculateElectiveAverage, calculateOverallAverage).

StudentSearcher implements Searchable

searchById, searchByName (case-insensitive substring match), searchByGradeRange, searchByType(StudentType) (fixed from a boolean — see Part IV). getSearchDescription() builds the human-readable label shown alongside results ("Name: \"John\"", "Type: Honors", etc.).

8.5 calculators

GPACalculator implements Calculable

Every method delegates to GpaLetterGrade — no independent thresholds of its own anymore. cumulativeGPA(studentId) averages percentageToGPA() across every grade; classRank(studentId) counts how many other students have a strictly higher average, +1.

StatisticsCalculator

The largest class in the project (253 lines). calculateDistribution() buckets every grade by LetterGrade.ordinal(). calculateStats() computes mean, median (average of the two middle values for an even count), mode (ties broken toward the lowest value via a TreeMap's ascending iteration order), standard deviation, and range, plus which student/subject earned the single highest and lowest grade. compareStudentTypes() now compares getType() instead of instanceof HonorsStudent (the same OCP fix as StudentSearcher).

8.6 export and imports

ReportGenerator implements Exportable

exportSummary(studentId) and exportDetailed(studentId) build report text as a StringBuilder; the detailed report maps each Grade to a GradeDTO via GradeMapper before formatting its grade-history table (the one DTO usage on the export side).

FileExporter

Wraps java.io.FileWriter directly — the one class in the project with no injected collaborator to mock, which is why it's the documented exception to the two-test-file convention (Part IX).

CSVParser

Parses a CSV, skipping (never aborting on) an invalid row — collects human-readable error strings instead. Validates, per row: exactly 4 columns, no empty field, a numeric grade in [0,100], a subject name that actually exists, and — the KI-10 fix — that the CSV's own declared SubjectType column actually matches the real subject's type (a row claiming "Elective" for a Core subject is rejected, where it used to import silently).

BulkImportService

Orchestrates: read file → CSVParser.parse() → for each valid row, look up the student (fail the row if unknown) → construct and persist a Grade → write a timestamped log file under imports/ summarizing the whole run, success or partial failure. A log-write failure is caught and logged, never silently swallowed, and never rolls back grades already recorded (the import itself already happened by that point).

8.7 console — Presentation Layer, and app — Entry Point

10 classes under console, one per menu option, all implementing MenuAction (getOptionNumber, getLabel, execute, isAuthorizedFor(Role) default true, terminatesLoop default false). ConsoleUtils holds the two behaviors more than one action needs: promptEnter(Scanner) and getAvailableStudentIds(StudentManager). Only AddStudentAction, ViewStudentsAction, RecordGradeAction, and BulkImportAction override isAuthorizedFor to require Role.TEACHER — every other action is readable by any role. These 10 classes have no automated tests of their own yet (0% in JaCoCo) — a known, deliberately scoped-out gap (Part XII), though each already takes its Scanner via constructor, so the same technique used for ConsoleApp below would work for each of them without further refactoring.

app.Main

The composition root, and nothing else. Builds every repository/service/manager/calculator exactly once as local variables, builds the List<MenuAction> in menu order, then hands both to a new ConsoleApp and calls run(). About 65 lines; deliberately too thin to need a test of its own.

app.ConsoleApp

The actual menu loop, extracted out of the old single-file Main: print menu (filtered by role) → read a number → find the matching action → check authorization → execute() inside a try/catch chain ending in ApplicationException. Takes its Scanner via constructor rather than binding to System.in in a static final field — that one change is what makes it unit-testable. Also had to move out of the default/unnamed package the old Main lived in: Java doesn't allow a class in a named package (like tests.app) to import a class from the unnamed package at all, so no test could ever have reached it there regardless of how testable its internals were. Covered by tests/app/ConsoleAppTest.java (scripted Scanner input, captured System.out, real MenuAction test doubles) and tests/app/ConsoleAppMockitoTest.java (mocked MenuActions, verifying dispatch/authorization/retry interactions) — 17 tests covering every exception-translation branch, the role-based menu filter, invalid-input handling, and the retry-on-InvalidGradeException flow.

8.8 dto / mapper, utils

StudentDTO/GradeDTO are plain immutable data holders (final fields, constructor, getters, no behavior) used only by Search Students and the exported detailed report — Add Student/Record Grade still use real domain objects, since they need the full validation surface a DTO intentionally doesn't carry. StudentMapper.toDto(Student): StudentDTO and GradeMapper.toDto(Grade): GradeDTO are the only two mapping methods in the project — one direction only (see Part VIII's earlier discussion on why there's no RequestDTO/ResponseDTO split: there's no HTTP request/response boundary in a console app).

utils.InputSanitizer.sanitize(String) trims and strips control characters from raw console input. utils.DateFormats centralizes five named SimpleDateFormat patterns that used to be copy-pasted independently into four different classes. utils.validators.StudentValidator/SubjectValidator are stateless static validators throwing the appropriate *ValidationException on the first rule broken.

9.1.1 Running JaCoCo and SonarQube

JaCoCo needs no separate setup — it attaches to the JVM automatically during the build:

mvn clean verify

Produces target/site/jacoco/index.html (a browsable per-package/per-class report) and target/site/jacoco/jacoco.csv (the same data, scriptable). The `report` execution is bound to Maven's `verify` phase, so `mvn test` alone does not regenerate it — `verify` (which runs `test` first anyway) does.

SonarQube needs a running server to send results to. Locally, on Windows, after downloading the free Community Edition:

# 1. Start the local server (first run initializes its own embedded DB - be patient)
bin\windows-x86-64\StartSonar.bat
# wait for "SonarQube is operational" in the log, then open http://localhost:9000
# (default admin/admin, forces a password change; create a project + token in the UI)

# 2. Run the analysis - clean verify first, in the SAME command, so Sonar's
# Java analyzer can read the JaCoCo execution data it needs for coverage:
mvn clean verify sonar:sonar -Dsonar.projectKey=<key> -Dsonar.token=<token>

# 3. Stop the server when done:
bin\windows-x86-64\StopSonar.bat

The sonar-maven-plugin is registered in pom.xml but bound to no lifecycle phase, so its mere presence never changes what mvn test/mvn verify do — `sonar:sonar` only runs when explicitly invoked.

9.2 The Two-File-Per-Class Convention

Nearly every production class has two test files:

  • <Class>Test.java — behavioral tests using real collaborators (or, for service/manager layers, a hand-wired real dependency chain all the way down to the in-memory repositories). Answers: "does this produce the right output?"
  • <Class>MockitoTest.java — the same class verified through mocked collaborators: interaction verification (did it call the repository exactly once?), call-order verification, and edge cases the real implementation can never actually produce but defensive code still guards against (a repository somehow returning null). Answers: "does this call its collaborators correctly, in the right order, the right number of times?"

Documented exceptions (no Mockito pair, because there's genuinely nothing to mock): LetterGradeTest, GpaLetterGradeTest (pure enum lookup functions), InputSanitizerTest, DateFormatsTest (pure static utilities), StudentMapperTest, GradeMapperTest (pure mapping functions over a real object, no branching to isolate), FileExporterTest (wraps java.io.FileWriter directly with no injected collaborator of its own).

9.3 JUnit 5 Lifecycle, and Every Annotation Actually Used

How JUnit 5 actually runs a test class (this is the part an engineer, not just a test-writer, should be able to explain): for every single @Test method, JUnit 5 creates a brand-new instance of the test class via reflection — this is the default PER_METHOD test instance lifecycle. That's why none of these 49 test files need to manually reset instance fields between tests: there is no shared state to reset, because no state is actually shared — each test method effectively gets its own object. (JUnit 5 also offers a PER_CLASS lifecycle via @TestInstance(Lifecycle.PER_CLASS), which reuses one instance for every test in the class — not used anywhere in this suite, since nothing here needs the one scenario that justifies it: sharing an expensive piece of setup, like a real database connection, across many tests.)

9.5 Mockito — Every API Actually Used, and How It Works Under the Hood

any() / anyString()10 / 1Argument matchers meaning "match this call no matter what was passed here" - needed once any argument in a stub/verify call uses a matcher, since Mockito then requires every argument on that call to be a matcher toowhen(action.isAuthorizedFor(any())).thenReturn(authorized);
spy(realObject)6Wraps a real object; every call runs the real method by default, but individual calls can still be verified or selectively overriddenRegularStudentMockitoTest spies on a real RegularStudent to confirm isPassing() genuinely delegates to calculateAverageGrade()/getPassingGrade() rather than duplicating that logic
doThrow(...).doNothing().when(mock).voidMethod()1 eachThe required syntax for stubbing a void method (there's no return value to hang when(...).thenThrow(...) off of) - also sequences different behavior across successive calls: throw the first time, do nothing the secondConsoleAppMockitoTest's retry test: doThrow(new InvalidGradeException(...)).doNothing().when(recordGrade).execute(); models "fails once, then succeeds on retry"
inOrder(mock1, mock2, ...) / InOrder3Plain verify() only checks count, never relative order. InOrder additionally asserts the calls happened in a specific sequence across one or more mocks - needed whenever the order of operations is itself part of the behavior under testGradeServiceImplMockitoTest: inOrder.verify(students).findStudentById(...); inOrder.verify(subjects).findSubjectByCode(...); inOrder.verify(grades).addGrade(...); - proves the student is checked before the subject, before the grade is ever persisted

How mock() actually works: Mockito's default "inline" mock maker (used automatically since Mockito 5, which this project depends on) uses the ByteBuddy library plus the JVM's java.lang.instrument Instrumentation API to redefine the target class's bytecode at runtime, inserting an interceptor into every method. When a mock's method is called, that interceptor consults an internal invocation log: if a matching when(...) stub was registered, it returns/throws whatever was stubbed; otherwise it returns Mockito's default "smart null" and records the call so a later verify(...) can look it up. This is also exactly what makes mocking a final class or method possible without a separate tool like PowerMock — the older "subclass proxy" mock maker couldn't override final methods (you can't override final via normal subclassing), but rewriting the class's own bytecode directly has no such restriction.

A real version-compatibility bug hit during this project: mockito-core:5.11.0 failed under JDK 25 with "Could not modify all classes" when mocking classes involving Student. Root cause: JDK 25 tightened restrictions on the exact bytecode instrumentation this inline mock maker relies on. Fixed by upgrading to mockito-core:5.23.0, a version with updated JDK-25 support. Good interview answer if asked "have you hit a real dependency/tooling bug?"

9.6 Testing Alternatives Not Used In This Codebase, and Why

AlternativeWhat it would do insteadWhy it wasn't used here
@Mock / @InjectMocks / @ExtendWith(MockitoExtension.class)The annotation-driven Mockito style: decorate test fields with @Mock, let the Mockito JUnit 5 extension instantiate and inject them automatically, and decorate the class-under-test field with @InjectMocks to have Mockito wire the mocks into its constructor for youEvery mock in this suite is instead created with an explicit mock(Class.class) call inside the test method body, and wired into the class-under-test by calling its real constructor directly. Part X — Git & GitHub Workflow

10.1 Branching Strategy

Strict main / develop / feature/* flow, matching the brief's required strategy exactly:

  • main — production-ready, protected; never touched directly during this entire project. Every fix and feature landed on develop first, with main gated for a later, separate authorization.
  • develop — the integration branch; every feature branch merges here via git merge --no-ff (never a fast-forward, so the merge commit itself documents the feature as a discrete unit in history).
  • feature/* — one branch per unit of work: feature/exception-handling, feature/export-report, feature/gpa-calculation, feature/bulk-import, feature/class-statistics, feature/search-students, feature/ci-cd, plus later feature/BugFix-professional-structure, feature/BugFix-solid-principles, feature/BugFix-package-structure, feature/BugFix-student-read-only, feature/BugFix-menu-role-filter, feature/BugFix-pom-jacoco-placement, and others — 16+ feature branches total across the project's life.

10.2 Commit Convention

Conventional Commits format throughout the later half of the project: feat(scope): ..., fix(scope): ..., refactor(scope): ..., test(scope): ..., docs(scope): .... This was itself one of the 14 issues found in a requirements review (only ~11% of early commits followed the convention) and explicitly fixed going forward — a good example to cite if asked "how do you handle discovering your own past mistakes."

10.3 The Required Workflow, As Actually Practiced

  1. Branch from develop
  2. Implement + commit incrementally, with tests added alongside the same commit or a following one
  3. Push the branch, let GitHub Actions CI run automatically
  4. Only once CI is green, merge (--no-ff) into develop locally
  5. Push develop, confirm CI is also green on develop post-merge (catches any merge-specific breakage a clean feature-branch CI run wouldn't)
  6. main stays untouched pending separate, explicit authorization

Part XIII — Anticipated Interview Questions (from the Assignment Rubric)

Drawn directly from ReadMe-v2.md's grading rubric and testing scenarios — these are close to exactly what a reviewing engineer would ask, since the rubric is their checklist.

Q: Walk me through what happens, end to end, when I record a grade.

A: console.RecordGradeAction.execute() reads and sanitizes the student ID, subject choice, and grade value via Scanner/InputSanitizer, then calls gradeManager.addGrade(grade). GradeManager delegates to GradeService.recordGrade(), which checks the student and subject both exist (throwing StudentNotFoundException/SubjectNotFoundException otherwise), then persists via GradeRepository.addGrade() into the Grade[200] array. The Grade object itself already validated its 0–100 range in its own constructor before any of this ran. Control returns up through the same call chain to ConsoleApp's loop.

Q: Where exactly is SOLID actually demonstrated, not just claimed?

A: See Part IV in full — but the strongest answer is naming the two real violations that were found and fixed during a dedicated audit (the Main God Class for SRP, and the boolean/instanceof-based student-type dispatch for OCP), not just reciting where each principle theoretically applies.

Q: How is a Student prevented from adding, editing, or deleting anything?

A: Every write-capable MenuAction (AddStudentAction, ViewStudentsAction, RecordGradeAction, BulkImportAction) overrides isAuthorizedFor(Role) to require TEACHER; every read-only action accepts the interface's default true. Enforced twice: the menu itself is filtered by role before printing, and the dispatch loop re-checks authorization before calling execute(), so a Student typing an unlisted option number directly still fails safely.

Q: What's the difference between your two test files per class?

A: <Class>Test uses real collaborators end-to-end and checks output correctness. <Class>MockitoTest mocks the collaborators and verifies interaction — that the right method was called, the right number of times, in the right order — plus edge cases the real implementation can't produce but defensive code still guards.

Q: Why doesn't this project use checked exceptions?

A: Every custom exception represents a business-rule violation with nowhere useful to recover locally — it needs to bubble to one central handler in ConsoleApp. Checked exceptions would force every intermediate method signature to declare throws, pure boilerplate with no benefit here.

Q: How would this scale to multiple concurrent users?

A: Honestly — it currently wouldn't, safely. The static ID counters (studentCounter, gradeCounter) and the array-backed repositories have no synchronization; concurrent writes would race. Moving to a server model would require AtomicInteger counters (or a database sequence) and either synchronized access to the repositories or a real concurrent data store — this is flagged honestly in Part XII, not hidden.

Q: Why did you build a custom Logger instead of using SLF4J/Log4j?

A: A single-threaded console app with one output stream and four levels doesn't need log rotation, multiple appenders, or MDC — the full weight of a logging framework. The hand-rolled version gives exactly what's used, zero dependencies, and would be a mechanical swap to SLF4J's facade later if this became a server app.

Q: What would you do differently if you started over?

A: Main itself already went through exactly this — its menu loop was extracted into app.ConsoleApp, which takes its Scanner via constructor instead of binding to System.in, and is now fully covered by 17 tests. The same technique just hasn't been applied to each of the 10 individual console/*Action classes yet, even though each already takes a Scanner the same way — that's the one remaining, well-understood coverage gap in an otherwise 92%+-covered codebase.

Part XIV — Conclusion

This project took a flat, 9-class in-memory console app and rebuilt it into a layered, SOLID-compliant, dependency-injected, exception-safe, 92%+ business-logic-tested Java application, with a genuine Maven build, a genuine CI/CD pipeline, and a Git history that actually followed the required feature-branch discipline rather than just claiming to. Every architectural decision documented here — array storage, the composition root, colocated interfaces, the two-file test convention, the unchecked exception hierarchy — has a specific, defensible reason behind it, and every known gap (console test coverage, single-threaded assumptions, O(n) lookups) is named rather than hidden. That combination — knowing not just what the code does but why it's shaped that way and where it would need to change — is what this report was built to put in front of a reviewing engineer.

Appendix C — Glossary (Quick-Reference Definitions)

TermDefinition
JVMJava Virtual Machine — the runtime that loads compiled .class bytecode and executes it, providing memory management (heap, GC), a call stack per thread, and platform independence
Metaspace / Method AreaWhere class metadata and static fields live — one copy per class, for the life of the JVM process
HeapWhere every object created with new lives; managed by the garbage collector
Stack frameOne method call's local variables + return address; popped when the method returns or an exception unwinds past it
BytecodeThe platform-independent instruction set the javac compiler emits and the JVM executes (e.g. invokevirtual, invokeinterface)
Dynamic dispatchResolving which method body actually runs based on the object's real runtime class, not its declared reference type — the mechanism behind polymorphism
Type erasureGeneric type parameters (<T>) exist only at compile time; the compiled bytecode has no record of them
Checked vs. unchecked exceptionChecked: compiler forces a throws declaration or a catch. Unchecked (extends RuntimeException): no such enforcement — used exclusively in this project
Happens-beforeA Java Memory Model guarantee that one thread's write becomes visible to another thread's subsequent read — what volatile/locks/synchronized establish
Composition rootThe single place in an application where the full object graph is constructed and wired together — here, Main
Mock vs. spyA mock replaces a collaborator entirely (every method is fake unless stubbed); a spy wraps a real object (every method is real unless overridden) — both from Mockito
CI/CDContinuous Integration / Continuous Delivery — automatically building and testing every push (here: GitHub Actions running mvn test) so integration problems surface immediately, not at release time
