# Lab 2: AI-Assisted Technical Documentation

**Subject project:** [student-grade-management](../../student-grade-management/) — a layered Java
console app (`model → repository → service → manager → Main`) for tracking students, subjects, and
grades, backed by fixed-size in-memory arrays.

## Task

Produce API/technical documentation for the backend that a new engineer could use to work on the
codebase without reading every file first — what each layer does, what every public class/method
does, the validation rules that gate every write, and the failure modes (exceptions) callers need
to handle.

The project already has a `docs/PROJECT_GUIDE.md` covering high-level architecture and folder
layout. This lab deliberately does **not** repeat that — [API_DOCUMENTATION.md](API_DOCUMENTATION.md)
is a method-level reference (signatures, preconditions, validation rules, thrown exceptions) that
didn't exist yet, generated by reading every class under `src/` rather than paraphrasing the
existing guide.

## Approach

1. Read every source file under `student-grade-management/src/` directly (not just the docs) —
   the whole point of "AI-assisted" documentation is that the AI reads the ground truth, not that
   it summarizes another document.
2. Prompted iteratively (see [prompts.md](prompts.md)) — starting from a generic "document this
   project" ask, then adding role framing, explicit source grounding, an output schema, and a
   verification instruction to cross-check the existing docs/README against actual code behavior.
3. That verification step is what surfaced two real discrepancies between the documented behavior
   and the implemented behavior (see "Known Issues" in the deliverable) — they're kept in the
   final doc rather than silently fixed, since documenting reality accurately is the deliverable.
